## Свойства слоёв

### 1. Каждый слой — самодостаточный продукт
Можно остановиться на слое 1 и иметь рабочий безопасный коннектор.
Можно остановиться на слое 2 и иметь поисковый коннектор. Можно дойти до
слоя 5 и иметь AI-агента. Все три состояния — валидные конечные точки.

### 2. Слои аддитивные, не замещающие
Слой 4 (regex-теги) не заменяет слой 5 (AI-классификация) — они работают
параллельно. Теги по правилам дают мгновенные категории, AI даёт смысловые
ярлыки. Два слоя поверх одного индекса.

### 3. Зависимости только вниз
Слой 3 нужен слой 2 (каталог вложений хранится рядом с индексом). Слой 5
нужен слой 2 (AI работает по индексу). Никто никому не нужен «вбок».

### 4. Пользователь выбирает свою высоту
Один коллега доволен слоем 1 — чтением только. Другой собирает слой 2 для
быстрого поиска. Третий идёт до слоя 5. Не навязываем максимум.

### 5. Решения откладываются
В слое 2 не решаем «делать ли слой 5» — это вопрос после выпуска слоя 2.
Не нужно сейчас гадать про AI-провайдера в коде слоя 2.

## Слой 0 — Базовый мост ✓ Готово

**Версия:** v1
**Состояние:** Прошёл валидацию

10 MCP-инструментов (список папок, статус папки, список писем, получить

**Известные слабости** (переданы слою 1 на устранение):
- Любой клиент имел полный разрушительный доступ.

- Не было списка доверенных для отправки.
- Не было процедуры подтверждения.
- Нумерация писем через позицию — неустойчива при удалении.
- Не было журнала действий.

## Слой 1 — Безопасный фундамент ✓ Готово

**Версия:** v2.0.0
**Объём:** 8 этапов плюс усиление после ревью

Защитные примитивы, которые превращают слой 0 в раздаваемый продукт:

- Уровни доступа (контролируются переменной окружения).
- Процедура подтверждения (гибридная: интерфейс клиента → терминал).
- Список доверенных получателей (доверие при первом разрешении из папки
  «Отправленные», подписан криптографически).
- Журнал действий в формате JSON Lines без чувствительного содержимого.
- Защитные границы (очистка HTML, санитизация, маркеры тела письма).
- Эксплуатационные защиты (ограничения частоты, защищённые папки, список
  отправителей 2FA, защита от дублей).
- Распространение: один файл-бандл, точка входа `bin`, готовность к `npx`.

### Критерии выпуска слоя 1
1. `npm audit --omit=dev` показывает ноль уязвимостей.
2. Файл-бандл меньше 1,5 МБ.
3. `npx -y github:user/repo#v2.0.0` устанавливается на чистой машине.
4. README и инструкция установки покрывают четыре MCP-клиента.
5. Уровень «только чтение» по умолчанию, повышение работает.
6. Список доверенных работает: первичная заливка, криптоподпись, строгая
   проверка скрытых копий.
7. Процедура подтверждения работает в трёх вариантах.
8. Журнал пишет правильные поля и не пишет чувствительные данные.
9. Подписанная метка `v2.0.0` и выпуск на GitHub.

### Что слой 1 не делает
- Локальный индекс заголовков (это слой 2).
- Каталог вложений (это слой 3).
- Файл правил `patterns.yaml` (это слой 4).
- AI-классификация (это слой 5).
- Пул IMAP-соединений с IDLE (это слой 7).
- Интеграция с Яндекс.Диском (это слой 6).

## Слой 1.5 — PMLF (многоуровневый фильтр) ✓ Готово

**Версия:** v2.1.1
**Состояние:** Выпущен в мае 2026 года

Анализ содержимого исходящих писем поверх списка доверенных:

- **Модуль политики:** файл `risk-policy.json` подписан криптографически,
  защищён от подделки.
- **Сканер исходящих:** 11 категорий обнаружений (платёжные карты,
  банковские реквизиты России, государственные удостоверения личности,
  ключи API, секреты ~35 вендоров, крипто-кошельки, медицинские данные,
  пометки конфиденциальности, мульти-сигналы фраз вывоза, аномалии формы
  данных, демографические персональные данные).
- **Отслеживание провенанса:** только в памяти, никогда не сохраняется на
  диск. Ловит шаблон prompt-injection «только что прочитал → отправил».
- **Оценщик риска:** композитная оценка в баллах от всех сигналов с
  обоснованиями.
- **Адаптивное подтверждение по риску:** 4 уровня трения интерфейса —
  малое (как сейчас), среднее (расширенная подсказка), высокое (защита
  от вставки), блок (требуется отдельная команда подтверждения).
- **Конвейер отправки:** выделение цепочки этапов в отдельный модуль.
- **Метаданные доверия:** дата добавления, последнего использования,
  счётчик, источник; автоматическая миграция формата.
- **Расширения утилиты оператора:** `--policy`, `--recent`, `--list-trust`,
  `--revoke-trust`, `--high-risk-send`.

### Что слой 1.5 не делает
Никакого индексирования (слой 2), вложений (слой 3), AI-классификации
(слой 5), IMAP IDLE (слой 7), Яндекс.Диска (слой 6) — это отдельные слои.

### Триггер для перехода слой 1 → слой 1.5
Глубокое ревью выпуска v2.0.0 выявило защиту исходящего содержимого как
крупнейший пробел: список доверенных защищает «куда» уходит письмо, но не
«что» внутри. AI с инъекцией мог отправить ключ API или паспортные данные
доверенному получателю — слой 1 этого не заметил бы. Слой 1.5 закрывает.

## Слой 2 — Быстрый поисковый агент ✓ Готово

**Версия:** v2.6.0 → v2.8.0 (выпущен; НЕ v3.0.0)
**Состояние:** Выпущен в июне 2026 года. Реализован как локальный **JSON-индекс** без
нативных зависимостей (SQLite сломал бы single-bundle `npx github:`-установку).

### Что построено
Локальный JSON-индекс заголовков плюс новые MCP-инструменты:

- `yandex_search_fast(запрос)` — поиск по локальному индексу менее 5 мс.
- `yandex_get_thread(идентификатор)` — полное дерево по полям ссылок и
  ответов.
- `yandex_unanswered(старше_дней)` — письма без ответа от владельца.
- `yandex_stats()` — топ-отправители, распределение по папкам, месячная
  динамика.

Плюс команды утилиты:
```bash
npx yandex-mail-mcp index build
npx yandex-mail-mcp index update
npx yandex-mail-mcp index status
npx yandex-mail-mcp index drop
```

### Что индексируется
Только конверт письма (около 300–500 байт на письмо): уникальный
идентификатор сообщения, папка, отправитель, тема, дата, флаги, размер,
наличие вложений, идентификатор треда, время индексации.

5000 писем занимают около 2 МБ. Первичная сборка — около 30 секунд.

### Триггер для перехода слой 1.5 → слой 2
1. Слой 1.5 выпущен и две-четыре недели отзывов.
2. Реальный запрос от ≥1 коллеги «поиск медленный».
3. Подтверждение, что модель безопасности слоя 1.5 работает в продакшене.

### Критерии выпуска
1. `index build` создаёт JSON-индекс менее 5 МБ на 5000 писем.
2. `search_fast` отвечает менее 5 мс на типовых запросах.
3. Новые инструменты появляются только при наличии индекса.
4. Автоматическое возвращение к живому IMAP при промахе кеша.
5. Идентификатор сообщения — первичный ключ (нумерация позиции
   нестабильна).
6. Документация: когда индекс, когда живой запрос.

## Слой 3 — Каталог вложений ✓ Готово

**Версия:** v2.9.0 (выпущен; минорный, не v3.0.0 — аддитивный, ничего не ломает)
**Состояние:** Выпущен 2026-06-23. Живой гейт на реальном ящике: 0.53 мс (порог — 50 мс).

### Что построено
Каталог вложений через IMAP-структуру `BODYSTRUCTURE` (без скачивания тел
писем). Инструменты:

```
yandex_find_attachments(от?, имя_содержит?, mime?, начиная_с?, до?, limit, offset)
  → { всего, hits: [{идентификатор, тема, дата, отправитель, папка, uid,
      имя_файла, mime, размер_кб, occurrence_count, occurrences[], ...}] }
yandex_search_fast(..., has_attachments?)   — новый булев фильтр
```

### Жёсткое ограничение
**Только метаданные.** Никакого парсинга PDF/DOCX/текста без явного запроса
пользователя на конкретный документ.

### Дедуп — по имени и размеру (sha256 нет)
Живой проб подтвердил: Yandex mail IMAP **не отдаёт хеш** вложений (md5=null, sha256 нет).
Поэтому дедуп идёт по `(имя_в_нижнем_регистре, размер)` с `occurrence_count`. Контентный
хеш sha256 отложен в слой 6 (Яндекс.Диск отдаёт его нативно — там и появится мост по хешу).

### Критерии выпуска (выполнены)
1. ✓ `find_attachments(от='компания', mime='application/pdf')` за менее 50 мс (факт — 0.53 мс).
2. ✓ Дедуп по имени+размеру (один файл в разных письмах = одна запись с `occurrence_count`).
3. ✓ Каталог (`attachments.jsonl`) лежит рядом с индексом слоя 2, изоляция аккаунтов.

## Слой 4 — Персональная таксономия ◄ СЛЕДУЮЩИЙ (по спросу)

**Версия:** ~v2.10.0 (следующий минор; план)
**Состояние:** Разблокирован слоем 3 (наличие вложений теперь индексируется). Строим
только по реальному запросу — не на упреждение.

### Что строим
Пользовательские regex-правила в файле настроек:

- Правила тегов: «тема соответствует X → тег: срочно».
- Правила извлечения: «тема `/Счёт.*?(\d+)/` → извлечённое.сумма».
- Правила источников: «домен отправителя в [сбер, втб] → тег: банк».

Правила применяются при индексации к теме и домену отправителя. Никакое
тело письма не парсится. Никакого AI.

### Критерии выпуска
1. Файл правил загружается и валидируется при старте.
2. Все инструменты поиска (слой 2) принимают фильтр по тегам.
3. Чтение письма возвращает поля тегов и извлечённого.
4. Горячая перезагрузка файла правил без перезапуска.

## Слой 5 — Семантический агент (AI-классификация)

**Версия:** v4.0.0 (план)
**Состояние:** Запланирован, опциональный

### Что строим
Отложенный кеш AI-разбора. При получении письма первый раз —
классификация через AI, сохранение результата по идентификатору сообщения
навсегда.

Три варианта провайдера (пользователь выбирает):

**A. Запрос через клиента (рекомендуется).** Просит клиента выполнить
вызов модели на его стороне. Использует подписку клиента, никакого
отдельного ключа.

**B. Внешний ключ API.** Anthropic, OpenAI или локальная Ollama по выбору.

**C. Эвристики из слоя 4** покрывают 70–80% случаев — на самом деле
большинство пользователей до слоя 5 не дойдут.

### Что классифицируется
```json
{
  "тип": "счёт | задача | уведомление | переписка | другое",
  "сводка": "...",
  "теги": [...],
  "извлечённое": {...},
  "требуется_действие": true,
  "описание_действия": "..."
}
```

Тело письма **не сохраняется** в кеш — только структурированный вывод.

### Критерии выпуска
1. Запрос через клиента работает на Claude Desktop / Claude Code.
2. Внешний ключ работает на одном провайдере.
3. Кеш «идентификатор → классификация» сохраняется между запусками.
4. Можно вручную сбросить запись (`yandex_reclassify(идентификатор)`).

## Слой 6 — Объединение источников

**Версия:** v5.0.0 (концепция)

### Что строим
Связь вложений с Яндекс.Диском через sha256.

Яндекс.Диск отдаёт sha256 файлов нативно в метаданных. Если в каталоге
вложений слоя 3 есть sha256 и в Яндекс.Диске есть файл с тем же хешем —
агент знает «это один документ, доступный через два источника».

### Триггер для перехода
1. Слои 3 и 5 выпущены.
2. Запрос пользователя «свяжи почту с диском».
3. Не менее трёх сценариев, где нужно именно объединение.

## Слой 7 — Реактивный агент

**Версия:** v∞ (последний релиз)
**Состояние:** Отложен

### Что строим
IMAP IDLE плюс новый инструмент:

```
yandex_wait_for_email(папка, от?, тема_содержит?, таймаут_секунд)
```

Для агентных сценариев: «отправь Алисе → дождись её ответа → продолжи».

### Почему отложен
- Узкая ниша (редкие сценарии).
- Требует постоянного IMAP-соединения с пулом.
- Меняет архитектуру (но слой 1 уже подготовил оболочку).
- Имеет смысл только когда слой 5 реально работает с почтой семантически.

## Протокол перехода между слоями

Перед началом нового слоя:

1. **Проверка реальности.** Предыдущий слой выпущен и используется не
   менее двух недель в продакшене.
2. **Сигнал спроса.** Не менее одного запроса от пользователя «нужна
   фича следующего слоя».
3. **Соответствие миссии.** Новый слой ведёт к видению поискового агента
   (не побочный путь).
4. **Аудит технического долга.** В предыдущем слое нет блокирующих
   ошибок.
5. **Архитектурное ревью.** Точки расширения из предыдущих слоёв всё ещё
   валидны или обновлены.

Если хоть один пункт не выполнен — пауза, разбираемся.
