# Миссия проекта

> Источник истины для всех архитектурных и продуктовых решений.
> Если решение не соответствует миссии — миссия пересматривается до изменения
> кода, а не после.

## Формулировка миссии

**Превратить личную или корпоративную почту Яндекса в высокоэффективного
поискового AI-агента, к которому коллеги имеют безопасный делегированный
доступ через любой MCP-клиент.**

Не «MCP-мост к IMAP». Цель — система, которая знает почту лучше владельца
ящика и которой можно безопасно делегировать поиск, классификацию, отправку.

## Кому и от чего больно

| Кто | Что болит | Что даём |
|---|---|---|
| Владелец ящика | Тысячи писем, забываются договорённости, теряются документы | Поисковый агент с полным индексом метаданных |
| Коллега-пользователь | Установка через `npm install` — высокий порог входа | `npx -y github:user/repo#vX.Y.Z` плюс одна инструкция в чат |
| MCP-клиент (любой) | Каждый клиент по-своему трактует подсказки опасности | Защита на стороне сервера, доверять клиенту не нужно |
| AI-агент | Prompt-injection через тело письма | Маркеры границ + санитизация + список доверенных получателей |

## Что мы строим

### Сейчас (Layer 1.5, v2.1)
Безопасный коннектор, который коллеги могут поставить и использовать, не
боясь утечки данных или случайной отправки. Защита от:
- Prompt-injection через тело письма.
- Утечки данных через скрытые копии незнакомым адресам.
- Случайного разрушения (удаления, перемещения в системные папки).
- Утечки кодов двухфакторной аутентификации в контекст AI.
- Подмены IMAP/SMTP-серверов через переменные окружения.

### Дальше (Layer 2 и выше)
Эволюция в поисковый агент. Индекс заголовков → каталог вложений →
персональная таксономия → опциональная AI-классификация. Подробности — в
[ROADMAP.md](ROADMAP.md).

### Долгосрочно
**Унифицированный документ-индекс владельца.** Письма, вложения и Яндекс.Диск
связаны через хеш-суммы и идентификаторы сообщений. Агент видит «все
материалы по этому контрагенту» одной командой.

## Что мы НЕ строим (зафиксированный отрицательный объём)

| Что | Почему нет |
|---|---|
| Парсинг содержимого PDF/DOCX | Дублирует всю почту на диске. Метаданных filename + хеш достаточно для 90% сценариев. Если пользователь явно попросит прочитать документ — сделаем для конкретного файла. |
| Мультипользовательский режим | Один коннектор = один Яндекс-аккаунт. Несколько ящиков подключаются как несколько MCP-серверов. |
| Мобильные клиенты | Только Node.js на десктопе. |
| Календарь и контакты | Только почта. CalDAV/CardDAV — отдельный проект. |
| Шифрование тела (PGP) | Вне области проекта. |
| OAuth через браузер | Безголовый контекст. Пароль приложения — основной способ авторизации. |
| Фоновые демоны | До явного запроса. MCP-сервер остаётся инструментом, который запускает клиент, а не постоянно работающим процессом. |
| Граф сущностей и кластеризация | Метаданных, тегов и поиска хватает. До запроса не делаем. |
| Публикация в npm | После стабилизации. GitHub с привязкой к тегу — основной способ распространения. |

## Критерии успеха миссии

Миссия выполнена, когда:

1. **Доверие.** Коллега ставит командой `npx -y github:user/repo#vX.Y.Z`,
   добавляет три переменные окружения и работает с почтой через любой
   MCP-клиент — без инцидентов безопасности.
2. **Скорость.** Поиск типового запроса («все письма от X за март») — менее
   5 миллисекунд (Layer 2 и выше). Сейчас — 800–1500 миллисекунд через
   живой IMAP.
3. **Охват.** Агент находит документ не только по теме письма, но и по имени
   вложения, дате, отправителю (Layer 3 и выше).
4. **Безопасность.** За весь жизненный цикл — ноль случайных отправок, ноль
   утечек 2FA-кодов, ноль потерь данных из-за неверного удаления (все слои).

## Принципы, которые не пересматриваются ради удобства

### 1. Безопасно по умолчанию
- Уровень «только чтение» — по умолчанию. Опасные операции не появляются в
  списке инструментов без явного включения.
- Недоверенное тело письма всегда заворачивается в маркеры границ.
- Предупреждение о правах доступа к файлам — всегда на Unix.

### 2. Защита на стороне сервера
- Не полагаемся на подсказки опасности от клиента, на интерфейс
  подтверждения в клиенте, на «модель послушная и не нажмёт отправить».
- Каждое опасное действие имеет внеконтекстное подтверждение с проверкой
  на стороне сервера.

### 3. Совместимость с любым клиентом
- 95% функций работают с любым MCP-клиентом (Claude Desktop, Claude Code,
  Cursor, Codex и будущими).
- Опциональная AI-классификация (Layer 5) предлагает три варианта
  провайдера, не привязывает к одному.

### 4. Послойная поставка
- Каждый слой — самодостаточный готовый к выпуску продукт.
- Не дробим слой N между релизами.
- Не строим слой N+1, пока слой N не выпущен и не получил отзывы.

### 5. Минимум поверхности
- Не добавляем функциональность «потому что прикольно».
- Каждая фича проходит тест: «решает реальную боль пользователя или строит
  фундамент для слоя выше?»

## Фильтр решений

Перед любым архитектурным решением — три вопроса:

1. **Соответствие миссии.** Это приближает к поисковому AI-агенту или это
   побочная функциональность?
2. **Стоимость безопасности.** Какую новую поверхность атаки это открывает?
   Митигировано?
3. **Принадлежность слою.** Этот код принадлежит текущему слою или это
   преждевременная работа из верхнего слоя?

Если ответ на (1) — «нет» и на (3) — «верхний слой», то парковка в отложенные
задачи, а не реализация.

## Когда документ пересматривается

- При завершении мажорной версии — проверка, что миссия всё ещё верна.
- При обнаружении противоречия в реальной работе («мы делаем X, но миссия
  говорит не X — что меняется?»).
- По явному запросу владельца.

Не пересматривается на основе одного раздражения, одного пользовательского
запроса, одного «было бы прикольно».
